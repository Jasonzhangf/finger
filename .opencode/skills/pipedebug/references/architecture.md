# RouteCodex V2架构详解

## 4层流水线架构

### 1. LLM Switch Workflow层 (核心中的核心)
**职责**: 统一的工具处理入口
- **工具规范化器**: 标准化工具格式，去除冗余
- **文本收割器**: 提取和清理文本内容
- **系统工具指引**: 注入行为标准化指引

**关键原则**:
- SSE、格式转换、工具处理都在此层统一处理
- 是整个系统的唯一工具处理入口

### 2. Compatibility层 (兼容性处理)
**职责**: provider相关的标准化处理
- 字段映射和转换
- 配置驱动的兼容性规则
- 错误兼容处理

**约束**:
- 只处理provider相关的标准化
- 不做多余转换
- 不处理工具调用

### 3. Provider层 (通信层)
**职责**: 与外部AI服务的通信
- 双向HTTP通信
- 认证和连接管理
- 模型ID双向替换

**核心功能**:
- 发送前替换为provider当前设置的模型ID
- 响应时替换回原始模型ID
- 唯一做通信和模型替换

### 4. External AI Services层
**职责**: 外部AI服务接口
- OpenAI、Anthropic、GLM、Qwen等
- 标准化API调用
- 错误处理和重试

## 数据流向

```
HTTP请求 → LLM Switch Workflow → Compatibility → Provider → External AI Services
         ↑                                            ↓
         ←──────────────── 响应流 ←─────────────────
```

## 关键设计原则

1. **Fail Fast**: 错误直接暴露，无隐藏fallback
2. **配置驱动**: 完全基于配置文件，无硬编码
3. **职责单一**: 每层只做自己的事
4. **双向处理**: 请求和响应都要经过完整的4层处理
